CAP theorem - 
    Diagram
    Brewer's theorem
    Consistency, Availability, Partition tolerance

    So we know ACID (Atomicity, Consistency, Isolation, Durable) has many limitations in distributed systems,
    ACID properties are weakened here, to survive CAP

    Statement : It is not possible to guarantee all three of the desirable properties - consistency, Availability and Partition tolerance
    at the same in a distributed system with data replication.

    CAP theorem is a constraint on distributed systems with replication
    Only 2 at a time, one has to be compromised
    eg - Banks : Consistency is more important, they compromise Availability (server downtimes)
        Facebook/insta : value Availability, work with eventual consistency

    combinations -
        CP - 
            Prioritize C and P over A 
            ensures that all nodes in system, have same consistent view of data,
            even in presence of network partitions
            If Partition occurs and a minority of nodes are isolated, the system sacrifices Availability (of isolated nodes) unitl the partition is resolved
            eg : banking transaction - all systems must agree on new account balances, before transaction is complete

        AP - 
            Focus on providing high A and P, even at the expense of sacrificing strong C
            Prioritize responsives and ensure that system remains available to clients, even if it means allowing temporary inconsistencies in data
            may use eventual consistency    
            eg: Social media feeds (pfp can come late, but insta shouldnt shut down), e-com product catalogs
        CA - 
            balance between C and A 
            aim to provide C and A, but not at same level as CP or AP systems
            more like traditional systems, follows ACID
            its not realistic in distributed systems as it assumes no paritions
            eg : smaller systems where partition highly unlikely



Base Theorem - Basically Available Soft State Eventually consistent
    Eric Brewer 
    it is a design approach for AP systems, it is how you implement an AP system under CAP 

    BASICALLY AVAILABLE means system will be available in terms of CAP
    SOFT State means even if no input provided, state will change over time i.e eventual consistency,
    which means, system will attain consistency eventually, if no input is sent during that time

    CA - Not Base, Strong consistency, High Availability
    CP - Not BASE, Strong C, may sacrifice A 
    AP - BASE fits, eventual C, always available
    thus, base is implementation of AP, its not connected to the other two



NRW Notations - 
    implementation of Eventual Consistency may vary accorss different NoSql DBs 
    these notations are used to describe how eventual consistency model is implemented accross NoSQL DBs where - 
        N is number of data copies that the DB maintains
        R is number of copies an application needs to refer to, before returning output of read request
        W is number of copies that need to be written to, before write operation is marked completed

    Write Operations- 
        N = W : updates all data copies before returning control to client, and marking as succesfull
            slows down performance (hight latency)
        W = 1, R = N : updates any one copy amd marks write as succesfull
            whenever read req, it will read all copies to return result
            very fast writes
        N > W > 1 : succesfull write to be done to nodes greater than one, but not all at same time 
            reduces probability of reading outdated data, while ensuring efficient writing

    Read Operations-
        R = 1 : reads any one copy, which can be outdated
        R > 1 : more than one copy is read, but slows down read operation (timestamp based)
        N < W+R : read operation in such case retrieves latest value
    
    Eg - 
        whatsapp profile pic updated, some friends saw it immediately, some see old one for few seconds
        if whatsapp has 3 servers (N = 3)
        Q if W = 1 , what might users observe immediately after update
        -> less consistency, some may see update, some may not
        Q what would change if W = 3
        -> slow updates but better consistency
        Q which gives better speed , and which gives better consistency
        -> speed : W=1, consistency : W = 3

        Order placed on amazon, on app they see confirmedm but website shows pending for few mins 
        amazon may use N = 5 replicas
        Q what would be the W, that causes delay ?
        -> W = 1 (R maybe 1 too) 
            but, why not W = 5 (like, writing to 5 may take time and hence delay)
            > cause, this would just cause latency for update accross all users, 
            our problem is about some seing update, some not, so W would be 1 to cause that 
        Q would you choose W = 3 or 1? why ?
        -> w =3, as that would be balance (N > w > 1) which leads to R = 3 , thus N < W + R
        Q if customers must always see correct status, what should be R (high or low)?
        -> High , R = w, or maybe high enough such that W + R > N

    Replication - 
    diagram
        Process that ens
        ures, same data is available on more than 1 mongoDB server
        required for increasing data availability

        Fault Tolerance - If the Primary server fails, data is still accessible from another server.
        Load Balancing - Distributes "Read" traffic across multiple servers so single server doesn't get overwhelmed.

        MongoDB groups servers into Replica Sets, which has - 
            Primary Server - handles all Write ops (insert, update, delete) from clients (can Read as well)
            Secondary Servers (worker) - can be used for all Read operations to balance the load
                                replicates data from primary


    Oplog (operation log) of primary - 
        used by primary server to record all changes to its data sets
        oplog is a capped collection and has timestamps
        secondary replicates this oplog and applies these changes to its own data sets (asynchronously)

    Heartbeat - 
        members in replica set send heatbeats to each other every 2 seconds
        if no heartbeat for 10 seconds from one member, others mark this member as inaccessible

    Arbiter - 
    diagram
        special secondary server, doesnt store any data
        only used for election process (voting process for changing primary server)
        used when cost prevents adding a full Secondary node but you need an extra vote for elections (tie-breaker)
        an arbiter always stays an arbiter, unlike primary/secondary nodes which can switch

    Voting - 
        Replica Set Limit: Can have up to 50 members total.
        voting members - only 7 members (including primary) can vote
        non-voting - rest 43 are read only nodes that cant vote

Sharding - split the data
diagram
    need - 
        huge volume of data in modern systems
        single server becomes slow and overloaded
        sharding helps distribute data accross multiple servers
        gives scalability and high performance
    
    sharding - 
    arch diagram
        horizontal partitioning of data
        large collection is split across multiple servers called Shards
        Each shard stores only a portion of data
        All shards together represent the full dataset

        shard - stores subset of data
        config server - Stores metadata (information about how data is distributed across shards)
        mongos (Query Router) - interface between client application and shards
            directs queries to correct shard, based on data request

    Steps - 
        client -> send query to mongos -> asks config server -> says "shard2" -> 
        mongos -> sends query to shard2 -> result sent to client
    
    Shard key - 
        field that decides how data is distributed
        should have -
            high cardinality, needs many unique values (user_id better than gender)
            evenly distribute data, to avoid overloading a single shard
        
        bad shard key (timestamp for a high-traffic app) causes all writes to hit one shard,
        creating a performance bottleneck (Hotspot problem)

        techniques - 
            Range-based – data divided into continuous ranges
                eg Shard A gets IDs 1-1000, Shard B gets IDs 1001-2000

                but leads to hotspots if data is inserted in a sorted order 
                    like dates
                    shard A - Jan 1to10, shard B - Jan 2to20 ...
                    On Jan 2, 100% of the traffic goes to Shard A
                    Shard A becomes a "Hotspot" because it's overloaded, while other shards sit idle
                    

            Hash-based – hash function is applied to the shard key value, to determine the destination shard
                distributes users randomly, but evenly across all shards
                not efficient for range-based queries as related data is scattered
        
each shard is a replica set (data inside shard is replicated)
together provides scalability + fault tolerance

Hadoop -   

example
    Day by day stock information for several symbols for several years
    Problem - Find the Maximum Closing Price for each stock symbol in a 1 TB dataset
    Bottleneck - 
        Read Speed: At 122 MB/sec, it takes 2 hours 22 mins just to read the file from the disk.
        Total Time: Reading (142 min) + Computing (60 min) + Network lag ... approx= 3+ Hours

    Hadoop solves this using two main principles: Divide & Conquer and Data Locality
    Parallel Processing - 
        Split the 1TB file into 100 small blocks (10GB each)
        Process all $100$ blocks parallelly on $100$ different nodes

        Read Time: Drops from 150 mins to < 2 mins (150/100)
        Computation Time: Drops from 60 mins to < 1 min (60/100)
    
    Data Locality - 
        Old Way: Move data over the network to a central CPU (slow)
        Hadoop Way: Move the computation (code) to the node, where the data (blocks) is already stored
        much faster to send a few kbs of code over a network, than gbs of data

example 2 - library with thousands of books spread across multiple
branches in different locations. You need to count the number of books by each author.

    method1 - 
        Bringing all books to one central branch to count them
        Central Collection: Physical movement of all books to one spot
        Counting: One person/team counts everything
        Returning: Moving all books back to original branches
        Extremely slow due to the time and effort of moving heavy data (books) across the network
    
    method2 - 
        Sending the "counting task" to each local branch
        Each branch counts its own books on-site (Computation where data lives)
        Aggregating Results: Branches send only the small summary (counts) to a central location
        Much faster because books stay put; only small, lightweight results move

Hadoop - 
    Framework for distributed processing of large datasets 
    accross clusters of commodity computers

    open source programming framework for storing large amount of data and doing computation
    based on java programming (C and shell scripts too)

    Focus - concept of moving computation close to storage for better performance
        reduces data transfer time by processing data on same node where its stored (data locality)

    Traditional - 
        uses Data transfer
        data stored in central location
        when computation to be d one, data has to be transferred to computation node
        time-consuming and bandwidth-intensive
    Hadoop - 
        uses Data locality
        splits large files into smaller blocks
        distributes them among different nodes (HDFS)
        each node has part of data, and computational resources to process it

    Data locality implementation-
        HDFS (Hadoop Distributed File systems) :
            splits large files into smaller blocks
            distributes them among different nodes in cluster
            replicates each block (3 replicas) to ensure fault tolerance
        MapReduce :
            When a job submitted, Hadoop's JobTracker tries to assign tasks to nodes that already hold the required data blocks
            if node has required data block, task runs, avoiding unnecessary data transfer
        
HDFS - 
    splits large files into smaller blocks
    distributes them among different nodes in cluster
    replicates each block (3 replicas) to ensure fault tolerance

    File system : 
        controls how data is stored and retrieved
        permissions, security and metadata
        manages space efficiently
        eg FAT32, NTFS, ext4

        doesnt have info of data from another machine
        doesnt have replication feature like that of HDFS

    Benefits - 
        supports parallel distributed processing
        handles failures (replicates blocks)
        scalable (can grow easily by adding more machines to the cluster)
        cost effective (commodity hardware used)
        If one node stops working, HDFS still works because copies of the blocks exist on other nodes.
        Immutable in Nature (files usually not changed after being written)


Architecture - 
diagram
    Hadoop has 2 nodes - Master/Primary/Name node & Data nodes

    Master Node - stores meta data of all nodes (never stores data), maintains DNs 
    Data Node - stores Actual data

    eg
        file "abc.txt" -> 500mb data
        split into 4 blocks - B1: 128, B2: 128, B3: 128, B4: 116
        (default block size in hadoop is 128mb)

        DN1     DN2     DN3     DN4
        B1      B2      B3      B4
        such mapping stored in master (IP address stored, so master knows whats stored where)
        client -> search abc.txt -> master -> searches & provides IP -> client

        if data stored directly as such, data loss possible
        thus store B1 copy in node 2,3,4 as well (replication)

    but if master goes down, hadoop has secondary name node
    Secondary Name node - 
        diagram
        metadata contains 
            FSIMAGE - current SS of the system (File structure ss)
            EDITLOG - all transactions like updates, deletes etc
            FSIMAGE reads EDITLOG and updates current state of system
        eg - 
            fsimage -> abc.txt and xyz.txt
            delete xyz.txt
            recorded in EDITLOG
            fsimage reads editlog and updates - fsimage iwhtout xyz.txt

        Secondary NameNode periodically requests EditLogs from the NameNode
        even Master contains such metadata (fsimage & editlogs); but with time, log sizes increases
            and makes master slower
        on failure, Master node updates its metadata from the FSIMAGE of secondary node

Racks - 
    collection of multiple servers
    diagram
        Suppose data comes from an IP around bangalore

    Rack Awareness Algo - 
        Each replica of a block to be stored on different rack
            so if 1 rack fails, we dont loose everything
        first replica in local rack (from where request came, eg bangalore)
        next replica on another remote rack in another datanode


Hadoop Read & Write -   
    Write - 
    diagram
        pipeline setup
            name node checks if file already exists, does user have permission, how many block needed
            selects DN based on rack awareness algo, avaialble space, load balancing
            sends client a list of avaialble DNs for each block
            DNs in that list connect to one another in a chain, to form a pipeline
        data streaming and replication
            client -> DN1 -> DN2 -> DN3 ... client sends to DN1
            DN1 sends to DN2 and so on, these DNs(3) write data of block sent by client (thus replication)
        pipeline shutdown (acknowledgement stage)
            DN3 -> DN2 -> DN1 -> client ... ack sent by each DN to client
            once client receives ack, block write is succesfull
            name node updates which data written where (metadata)

    Read - 
        Name node decides which DNs are free for read and client reads from those DNs
        NameNode returns the list of DNs that contain replicas of each block
        client chooses one of those DN replica (usually from nearest node) for each block

MapReduce - 
    computing paradigm for processing data residing on many computers
    used for distributed computation
    Traditional models work only on single machines; 
    MapReduce is required when data is kept on multiple machines in a distributed manner

    Both Map & Reduce work on (Key, Value) pairs

    Record Reader -
        input records are often not in (k,v) format
        Converts raw input into (K, V) pairs for the Mapper
            key: usually the position of line in file
            value: actual line
    
    Combiner (Optional "Mini-Reducer") - 
        function that summarizes Mapper output records, with the same key, before passing them to the Reducer
        reduces network congestion by reducing amount of intermediate data sent across the cluster
        runs after mapper and before reducer

        when run MapReduce job on large datasets, large intermediate data generated by Mapper
        this used to increase the network congestion, combiner solves this

        diagram (for understanding)

    
    eg -
        file1.txt -> 500mb (thousands of records)
        to find count of each word

        split into 4 blocks, each block on one node
        
        but, records arent (key,value) pairs
        passed to record reader which converts to (k,v) pairs
            key: 00, value:"Hello how are you" and so on
        these pairs will be passed to Mapper
            written by developer
            ignores keys, splits values based on spaces
            generates intermediate pairs like (Hello, 1), (how, 1), (are, 1), (Hello, 1)
        reducer - 
            written by developer
            iterate through the list of "1"s for each word and sum them up
            Resulting pairs like (Hello, 13), (how,  10) etc


    Block vs Input Split - 
        Physical partition of the file.
            Logical partition of the file
        contains actual raw data chunks (128 MB)
            contains information on block addresses and record boundaries (pointers)
        Fixed size; can cut a record in half if it doesnt fit
            Identifies where the first whole record begins and the last ends
            ensures a Mapper never receives a half-broken record
        Number of blocks != Number of tasks
            Number of Map tasks = Number of Input Splits
        Block 1 might end with "Hello Wo" and Block 2 starts with "rld"
            Input Splits fix this
